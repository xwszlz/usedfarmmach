## 一、背景与问题

### 1.1 用户明确的工作流程

```
小程序上传图片（AI识别已取消）
    ↓
网站产品发布页 → AI智能识别 → 一键补齐参数 → 发布
    ↓
产品详情页 → AI深度分析 → AI估值
```

### 1.2 当前三个断点

| 断点 | 位置 | 问题 | 影响 |
|------|------|------|------|
| **断点1** | `ai-assistant.tsx` L188 | 前端直调外部FC `shendiao-ai-api-ybemtmjsna.cn-beijing.fcapp.run/recognize` | 识别功能Failed to fetch |
| **断点2** | `deep-analysis-card.tsx` L45 | 前端直调外部FC `shendiao-ai-api-ybemtmjsna.cn-beijing.fcapp.run/deep-analysis` | 深度分析Failed to fetch |
| **断点3** | 识别 → 深度分析之间 | 无isChineseBrand传递，无数据衔接 | 两个阶段各自为战 |

### 1.3 内部API已就绪但未被使用

项目内部已有完整的API路由，包含多模型降级链，但前端从未调用：

| 内部API | 路径 | 模型链 | 状态 |
|---------|------|--------|------|
| 图片识别 | `/api/agents/seller-helper/recognize` | 豆包→Gemini→OpenRouter(5级) | ✅就绪，未被调用 |
| 深度分析 | `/api/agents/seller-helper/deep-analysis` | 豆包→Gemini→OpenRouter(3级) | ✅就绪，未被调用 |

## 二、完善方案总览

### 2.1 核心原则

1. **前端只调内部API** — 彻底弃用外部FC端点
2. **isChineseBrand贯穿全链路** — 从识别阶段就判定，后续自动路由
3. **三阶段双引擎对齐** — 识别/深度分析/估值三个阶段的国内国际分流保持一致
4. **识别结果传递到深度分析** — 发布时存储的识别数据在详情页可复用

### 2.2 全链路流程

```
卖家上传图片（网站）
    ↓
Stage 1: AI智能识别
    ├─ 国内农机 → 豆包引擎（中文铭牌/中文品牌名）
    └─ 国际农机 → Gemini引擎（英文铭牌/欧美品牌名）
    ↓
一键补齐参数 + 自动判定isChineseBrand
    ↓
产品发布 → 数据库（存储isChineseBrand + 识别置信度）
    ↓
产品详情页
    ↓
Stage 2: AI深度分析
    ├─ isChineseBrand=true → 国内引擎（豆包 + 补贴数据参照价）
    └─ isChineseBrand=false → 国际引擎（Gemini + 出口估值参照）
    ↓
Stage 3: AI估值
    ├─ isChineseBrand=true → V5补贴估值引擎
    └─ isChineseBrand=false → V4出口估值引擎
```

function determineChineseBrand(brand: string): boolean {
  const chineseBrands = [
    "东方红", "雷沃", "沃得", "大疆", "极飞", "久保田", "洋马", "井关",

### 3.3 Stage 3: 估值衔接

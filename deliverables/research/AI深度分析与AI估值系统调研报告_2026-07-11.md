> 调研时间：2026-07-11  
> 调研范围：usedfarmmach.com 的 AI深度分析 + AI估值V4 系统
> 调研目标：定位"Failed to fetch"根因，评估当前模型调度与估值方案，寻找更优免费模型方案

## 一、现状梳理

### 1.1 AI深度分析（Deep Analysis）

| 维度 | 现状 |
|------|------|
| 前端组件 | `src/components/product/deep-analysis-card.tsx` |
| 前端调用地址 | `https://shendiao-ai-api-ybemtmjsna.cn-beijing.fcapp.run/deep-analysis` |
| 后端（备用） | `src/app/api/agents/seller-helper/deep-analysis/route.ts` |
| 主模型 | 豆包（火山引擎 ARK） |
| 备用模型 | Gemini 2.5 Flash → OpenRouter `google/gemini-2.0-flash-001:free` |
| 输入 | 3张图片 + 可选视频（HTTP URL） |
| 输出 | Markdown报告 + 结构化JSON |
| 分析内容 | 设备识别、技术参数、操作维修、市场参考价、购买建议 |

**截图中的问题**：

`AI深度分析` 按钮点击后显示 **"Failed to fetch"**。根因已定位：

1. **前端没有调用 Next.js 自己的 API，而是直接请求阿里云函数计算（FC）外部地址**。
   - 外部 FC 端点 `shendiao-ai-api-ybemtmjsna.cn-beijing.fcapp.run` 可能已过期、未部署或冷启动。
2. 浏览器直接请求外部域名，存在 **CORS 跨域**、**网络超时**、**端点不可用** 等风险。
3. 项目内部其实已经有完整的 `src/app/api/agents/seller-helper/deep-analysis/route.ts` 实现，包含三级降级链，但**未被使用**。

> **结论**：前端改走 `/api/agents/seller-helper/deep-analysis` 即可解决大部分问题，同时统一模型调度。

### 1.2 AI估值V4（Valuation）

| 维度 | 现状 |
|------|------|
| API路由 | `src/app/api/valuation/route.ts`（POST + GET） |
| 前端组件 | `src/components/valuation/valuation-card.tsx` |
| 估值公式 | `src/lib/valuation/formulas.ts` |
| 品牌/品类基准价 | `src/lib/valuation/brand-data.ts` |
| 图片分析 | `src/lib/valuation/image-analyzer.ts` |
| 视频分析 | `src/lib/valuation/video-analyzer.ts` |

**估值公式（V4）**：

```
估值 = 基准价 × 品牌系数 × 年份因子 × 工时因子 × 成色因子 × 市场因子 × 规格因子 × 视频因子
```

| 因子 | 说明 | 数据来源 |
|------|------|----------|
| 基准价 | 品类默认价或型号精确价 | 手动维护的 `brand-data.ts` |
| 品牌系数 | 进口高端 > 1，国产 < 1 | 手动维护 |
| 年份因子 | 0-3年每年5%，4-7年每年6%，8-12年每年5%，13年+每年4% | 硬编码折旧表 |
| 工时因子 | 年均预期500h，低工时奖励5%，高工时惩罚最高15% | 硬编码 |
| 成色因子 | excellent/good/fair/poor 或视觉评分映射 | 视觉评分来自LLM |
| 市场因子 | 国际市场价对比，范围70%-130% | 数据库 `internationalPrices` |
| 规格因子 | 马力/驱动/配置/重量/尺寸加权 | 硬编码规则 |
| 视频因子 | 0.92~1.08，运转正常加分，异常扣分 | GPT-4o视频分析 |

**当前模型调用链**：

| 功能 | 主模型 | 降级模型 | 备注 |
|------|--------|----------|------|
| 图片分析（估值） | Google Gemini 1.5 Flash | OpenRouter Llama 3.2 11B Vision | 主模型免费 |
| 视频分析（估值） | OpenRouter GPT-4o | 无 | GPT-4o 付费，无降级 |
| 深度分析 | 豆包 doubao-1.5-vision-pro-32k | Gemini 2.5 Flash → OpenRouter | 主模型依赖FC |
| 买家客服 | OpenRouter GPT-4o-mini | OpenRouter Gemini 2.0 Flash | 付费/免费混合 |
| 图片识别 | 豆包 seed-evolving | Gemini 2.5 Flash → OpenRouter | 主要用于卖家发布 |

### 1.3 模型调度现状

**环境变量配置**（来自 `.env` / `.env.local`）：

| 变量 | 当前值 | 状态 |
|------|--------|------|
| `ARK_API_KEY` | 已配置 | 火山引擎豆包 |
| `ARK_MODEL_ID` | `doubao-seed-evolving` | 代码中 deep-analysis 默认是 `doubao-1.5-vision-pro-32k` |
| `GOOGLE_API_KEY` | 已配置 | Gemini 免费调用 |
| `OPENROUTER_API_KEY` | 已配置 | 付费/免费模型兜底 |

**调度问题**：

1. **调度路径不统一**：深度分析、估值图片、估值视频、客服、识别分别用不同的模型优先级。
2. **缺少统一模型路由层**：没有统一封装 `callLLM()`，各模块各自处理降级和重试。
3. **缺少成本/配额监控**：没有 API 调用计数、失败率、token 用量统计。
4. **视频分析只有 GPT-4o 一个选项**：一旦 OpenRouter 余额不足或网络异常，视频分析直接降级为默认结果，没有免费替代方案。
5. **深度分析前端绕过 Next.js**：维护两条并行路径，增加了失效风险。

## 三、问题根因与改进建议

### 3.1 紧急修复（P0）

| 问题 | 修复方案 | 影响 |
|------|----------|------|
| 深度分析 `Failed to fetch` | 前端 `deep-analysis-card.tsx` 改调用 `/api/agents/seller-helper/deep-analysis` | 解决截图问题，统一维护 |
| 深度分析前端直接调用外部FC | 弃用 `shendiao-ai-api-ybemtmjsna.cn-beijing.fcapp.run` | 避免CORS/冷启动/单点故障 |
| 视频分析无免费降级 | 为视频分析增加 Gemini 2.5 Flash / 豆包 视频能力作为降级 | 降低OpenRouter GPT-4o成本 |

### 3.2 架构升级（P1）

| 改进项 | 方案 | 收益 |
|--------|------|------|
| **统一模型调度层** | 新建 `src/lib/llm/model-router.ts` | 一处配置，全局复用；自动降级；统一日志 |
| **模型选择策略** | 按任务类型、成本、成功率动态选择 | 免费优先，付费兜底 |
| **API 调用监控** | 记录每次调用：模型、耗时、token、成本、状态码 | 可优化、可报警 |
| **估值 V3/V4 优化** | 补充更多型号基准价；引入 LLM 辅助参数补全 | 提升估值准确率 |
| **缓存层** | 相同产品图片的 LLM 分析结果缓存 24h | 减少重复调用，降本 |

## 五、结论

1. **AI深度分析失败的根本原因**：前端绕过 Next.js 内部 API，直接请求一个外部阿里云 FC 地址，导致 `Failed to fetch`。**最快修复是改走内部 API**。
2. **AI估值V4整体可用**：公式结构合理，但视频分析过度依赖付费的 GPT-4o，缺少免费降级。
3. **模型调度需要统一**：当前各模块各自为战，建议建立统一模型路由层，优先使用免费模型（豆包、Gemini），付费模型兜底。
4. **免费模型资源丰富**：豆包每日200万Token + Gemini 250次/天，足以支撑当前产品分析需求，无需额外付费。
